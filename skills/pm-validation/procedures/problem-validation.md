# Problem Validation

Use when testing whether a target customer has a real, frequent, painful, and valuable problem.

## Steps

1. Define target segment and problem statement.
2. Gather evidence of current behavior, workarounds, spend, urgency, and frequency.
3. Avoid validating solution interest too early.
4. Identify invalidation signals.
5. Define decision thresholds.
6. Recommend continue, narrow segment, reframe, or stop.

## Output

Return evidence, assumptions, confidence, invalidation signals, decision rule, and next action.

## Done when

- Target segment and problem statement are stated separately; evidence covers current behavior, workarounds, spend, urgency, and frequency, with each row tied to an evidence ID (interview, ticket, observation, transaction).
- At least three hypotheses about the problem (severity, frequency, segment fit) are written IF/THEN/BY, and invalidation signals that would falsify them are listed.
- A decision threshold (e.g., minimum number of corroborating signals, minimum spend/urgency level) is stated before interpretation; confidence is recorded per hypothesis.
- When solution interest is being substituted for problem evidence, or when no behavioral evidence exists, the procedure refuses to declare the problem validated and recommends continue, narrow segment, reframe, or stop with the next learning step.
